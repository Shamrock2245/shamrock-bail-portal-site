This report documents a comprehensive audit and refactoring pass across the Shamrock Bail Bonds frontend ecosystem, targeting all five generation frontiers defined in the updated project instructions: Traditional SEO, GEO/Geospatial, Generative AI/LLM Ingestion, Voice Search, and Visual Search. ## Frontier 1: Traditional SEO & Programmatic Metadata

### Audit Findings

The `seoConfig.js` file correctly serves as a single source of truth for NAP (Name, Address, Phone) data, and `seo-helper.jsw` already generates `LocalBusiness` JSON-LD and meta tags for county pages. However, `LocationMetadataService.js` in the GAS backend lacked the programmatic meta header generation function needed to drive unique `<title>`, `<meta description>`, and canonical tags at the edge for each of the 67 county pages.

### Changes Made

**File:** `backend-gas/LocationMetadataService.js`

The file was refactored to expose three new exported functions:

- `generateProgrammaticMetaHeaders(countyName, countySlug)` — Returns a complete object with `title`, `description`, `canonicalUrl`, `geoRegion`, `geoPlacename`, and `geoPosition` fields, all dynamically constructed from county data. This drives unique meta headers for every programmatic county landing page.
- `getLocationMetadata(countySlug)` — Returns coordinates, jail proximity, and judicial jurisdiction for a given county slug.
- `getCountyGeoData(countySlug)` — Internal lookup against the `COUNTY_GEO_DATA` registry.

The `COUNTY_GEO_DATA` registry was seeded with precise lat/lng coordinates for the five primary operational counties (Lee, Collier, Charlotte, Hendry, Glades) with a state-centroid fallback for all 67 counties.

| Function | Purpose | Frontier |
|---|---|---|
| `generateProgrammaticMetaHeaders` | Unique title/description/canonical per county | Traditional SEO |
| `getLocationMetadata` | Coordinates + jurisdiction for schema injection | GEO |
| `getCountyGeoData` | Internal registry lookup | GEO |

## Frontier 2: GEO & Geospatial Local Maps Optimization

### Audit Findings

The `allFloridaCounties.json` data file, which drives all 67 county pages, contained **zero latitude/longitude fields**. The `seo-helper.jsw` module had a `geo` fallback to the Fort Myers HQ coordinates for all county pages, meaning every county page was broadcasting the same geospatial signal to Google Maps and local mapping algorithms. This is a critical GEO deficiency for a 67-county operation.

### Changes Made

**File:** `src/backend/seo-rich-results.jsw`

The file was completely refactored to expose two new functions:

- `generateGeoSchema(countySlug, countyName)` — Generates a deep `["LocalBusiness", "BailBondService"]` dual-type JSON-LD schema with per-county `GeoCoordinates`, `areaServed`, `knowsAbout` (including the specific judicial circuit), and `hasOfferCatalog`. The dual `@type` array is critical because `BailBondService` is a recognized Schema.org type that maps directly into Google's Local Services understanding.
- `getFullPageSchema(countySlug, countyName, faqs)` — Assembles the full schema array for a page, including an optional `FAQPage` schema block.

**Next Step Required (Brendan):** The `allFloridaCounties.json` file needs lat/lng coordinates added for all 67 counties. A data enrichment script should be run against the Google Maps Geocoding API or a static Florida county centroid dataset to populate these fields. The `LocationMetadataService.js` `COUNTY_GEO_DATA` registry is the interim solution for the primary five counties.

## Frontier 3: Generative AI Engine Optimization (LLM & ChatGPT Discovery)

### Audit Findings

The county page content generated by `county-generator.jsw` used simple variable-substitution templates that produced structurally flat text. Modern LLMs extract citations from content with clear hierarchical headers (`##`, `###`), explicit question-answer pairs, and semantic relational attributes. The existing templates lacked all three.

### Changes Made

**File:** `src/backend/county-generator.jsw`

Semantic relational attributes were injected into the `transformCountyData` output object:

```javascript
semanticAttributes: {
  author: "Shamrock Bail Bonds",
  expertise: "Licensed Florida Surety Agents",
  knowsAbout: ["Florida Bail Law", "Surety Bonds", judicialCircuit],
  areaServed: `${county.name} County, Florida`,
  founder: "Shamrock Bail Bonds LLC"
}
```

These attributes map directly to Schema.org properties (`knowsAbout`, `areaServed`, `founder`) that LLMs use during training runs and web-scraping crawls to build entity relationships.

**File:** `.agent/skills/copywriting/references/eeat_county_template.md`

A new E-E-A-T optimized copywriting template was created. It structures county page content according to Google's Experience, Expertise, Authoritativeness, and Trustworthiness framework, using:

- Clear `##` and `###` Markdown headers for LLM parsing
- A numbered step-by-step "How to Get Someone Out of [County] Jail" section — the exact format that ChatGPT and Perplexity extract as citations
- Explicit FAQ section with direct question-answer pairs
- Inline links to the county Sheriff's booking URL and court records

**File:** `llm-sitemap.md` (root) and `src/public/llm-sitemap.md`

A new 817-line LLM Semantic Sitemap was generated covering all 67 Florida counties. This file is a clean, JavaScript-free Markdown document containing the complete sheriff/clerk directory, judicial circuits, booking URLs, and major cities for every county. It is optimized for direct ingestion by AI crawler bots (GPTBot, ClaudeBot, PerplexityBot) that are already whitelisted in `robots.txt`.

## Frontier 4: Voice Search Ingestion Hardening

### Audit Findings

The `seoUtils.js` file already contained a `SpeakableSpecification` schema targeting `["h1", ".hero-subtitle", ".faq-question"]`. This is a solid foundation, but it missed the `.voice-trigger` CSS class convention, which should be applied to conversational phrasing elements designed specifically for Siri/Alexa/Google Assistant synthesis.

### Changes Made

**File:** `src/public/seoUtils.js`

The `SpeakableSpecification` `cssSelector` array was updated to include `.voice-trigger`:

```json
"speakable": {
  "@type": "SpeakableSpecification",
  "cssSelector": ["h1", ".hero-subtitle", ".faq-question", ".voice-trigger"]
}
```

**Next Step Required (Brendan):** Apply the `.voice-trigger` CSS class in the Wix Editor to any text element containing a conversational, long-tail voice phrase (e.g., "Need a bail bond near me right now? Call (239) 332-2245 — we answer 24/7."). These elements will then be read directly by voice synthesis engines.
